기술 블로그 글을 분석하고, 이 프로젝트에 적용 가능한지 확인하고 적용 가능하면 이슈 생성까지 연결해줘.

블로그 글: $ARGUMENTS

## 진행 순서

### 1단계: 블로그 글 분석

URL이 주어진 경우 WebFetch로 내용을 읽고, 텍스트로 주어진 경우 그대로 분석한다.

아래 항목을 추출:

```
## 블로그 분석

**제목**: {블로그 제목}
**핵심 주제**: {1~2줄 요약}
**사용 기술**: {언급된 기술 스택}
**해결하는 문제**: {어떤 문제를 해결하는 내용인가}
**핵심 구현 패턴**: {코드 레벨 핵심 내용}
```

---

### 2단계: 프로젝트 적용 가능성 검토

현재 코드베이스를 실제로 탐색해서(Glob, Grep, Read 활용) 다음을 판단:

**체크리스트**
- [ ] 현재 기술 스택으로 구현 가능한가? (Spring Boot, MySQL, Redis, FCM, WebSocket)
- [ ] 이미 동일/유사한 기능이 구현되어 있지 않은가?
- [ ] 기숙사 서비스 맥락에서 실제로 유용한가?
- [ ] 구현 복잡도가 합리적인가? (과도한 인프라 없이 가능한가)

**출력 형식**:
```
## 적용 가능성 검토

**결론**: ✅ 적용 가능 / ⚠️ 부분 적용 / ❌ 적용 어려움

**이유**:
- {적용 가능/불가능 근거}
- {기존 코드와의 관계 (중복/보완/신규)}

**적용 범위** (가능한 경우):
- 적용할 도메인: {domain명}
- 구현 방향: {어떻게 구현할지 1~2줄}
- 예상 난이도: 낮음 / 보통 / 높음
```

적용이 어렵거나 의미 없는 경우, 이유를 설명하고 종료.

---

### 3단계: 더 나은 대안 비교 (선택적)

블로그에서 제안한 방법 외에 **이 프로젝트에 더 잘 맞는 대안**이 있으면 비교 제시:

```
## 대안 비교

| 방법 | 장점 | 단점 | 이 프로젝트 적합성 |
|------|------|------|-------------------|
| 블로그 방법 | ... | ... | ... |
| 대안 A | ... | ... | ... |
| 대안 B | ... | ... | ... |

**추천**: {어떤 방법이 왜 더 나은지}
```

대안이 없거나 블로그 방법이 최선이면 이 섹션 생략.

---

### 4단계: 테스트 방법 제시

구현 후 **검증 방법**을 구체적으로 제시:

```
## 테스트 방법

### 단위/통합 테스트
- 테스트 시나리오: {무엇을 검증해야 하는지}
- 사용 도구: @DataJpaTest / @SpringBootTest / Mockito / H2
- 핵심 검증 조건: {assert 조건}
- 예시 코드 스니펫: {간단한 테스트 코드}

### 부하 테스트 (필요한 경우)
- nGrinder 시나리오: {vUser 수, 요청 수, 측정 지표}
- Before/After 비교 기준: {어떤 메트릭으로 개선을 측정할지}
- MySQL 확인 쿼리: {SHOW STATUS 또는 performance_schema 쿼리}
```

---

### 5단계: 사용자 확인

다음을 한 번에 질문:

```
## 확인 사항

1. 이 기능을 프로젝트에 구현하시겠습니까?
   - 구현한다 → 이슈를 생성합니다
   - 구현하지 않는다 → 분석만 완료합니다

2. (대안이 있는 경우) 블로그 방법과 추천 대안 중 어떤 방식으로 구현할까요?
```

---

### 6단계: 이슈 생성 (구현 결정 시)

사용자가 구현을 결정하면 다음 정보를 확인:

**불명확한 항목만** 추가로 질문:
- **대상**: 어느 도메인에 적용?
- **범위**: 이번 이슈에서 할 것 / 안 할 것
- **우선순위**: 급한가, 나중에 해도 되는가

작업 목록 초안을 제안하고 확인 후 GitHub 이슈 생성:
- owner: Teach-D
- repo: appcenter-16.5-dormitory-project
- title: `[{type}] {기능 설명}`
- body: 개요 + 블로그 출처 + 작업 목록 + 테스트 방법

브랜치 생성 후 로컬 체크아웃:
```bash
git fetch origin
git checkout -b teach/{type}/{설명}-{이슈번호} origin/teach/{type}/{설명}-{이슈번호}
```

`.claude/issues.md` 업데이트 후 커밋 & 푸시.

---

### 7단계: 구현 진행 여부 확인

```
## 완료

- 이슈 #{번호}: {제목}
- 브랜치: teach/{type}/{설명}-{이슈번호} (체크아웃 완료)

지금 바로 구현을 시작할까요?
- 시작한다 → `/work`를 실행합니다
- 나중에 → 이슈만 생성된 상태로 유지합니다
```

사용자가 바로 시작을 원하면 `/work` 워크플로를 이어서 실행.
