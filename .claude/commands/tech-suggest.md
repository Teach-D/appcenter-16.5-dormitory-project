각 항목은 반드시 "문제 - 해결 - 성과" 3줄 구조로 이력서에 바로 쓸 수 있게 출력한다.

## 진행 순서

### 1단계: 기술 키워드 해석

입력된 기술/지식 키워드를 아래 관점으로 먼저 해석한다:

- **핵심 개념**: 이 기술이 해결하는 문제는 무엇인가?
- **Spring Boot 생태계 연관 기술**: 이 기술과 함께 자주 쓰이는 Spring 관련 컴포넌트/어노테이션은?
- **이 프로젝트 스택 관련성**: 현재 스택(Redis, QueryDSL, FCM, WebSocket 등)과 어떤 접점이 있는가?

키워드가 모호하거나 넓은 경우(예: "캐싱", "비동기", "보안"), 하위 개념을 열거하고 어떤 범위로 분석할지 먼저 명시한다.

---

### 2단계: 프로젝트 코드 탐색

**탐색 범위:**
- `src/main/java/com/example/appcenter_project/` 전체
- `src/main/resources/` (설정 파일, Flyway SQL)
- `build.gradle` (의존성 확인)

**탐색 방법:**
- 관련 어노테이션, 클래스명, 메서드명, 설정 키를 기반으로 검색
- 예: Redis → `@Cacheable`, `RedisTemplate`, `application.yml의 spring.data.redis`, `RedisConfig`
- 예: N+1 → `@EntityGraph`, `fetch join`, `@BatchSize`, `LazyInitializationException`
- 예: 비동기 → `@Async`, `AsyncConfig`, `CompletableFuture`

실제 파일을 열어서 확인한 내용만 기반으로 분석한다. 추측으로 작성하지 않는다.

---

### 3단계: 결과 출력

**모든 항목(이미 적용 + 추가 가능)은 아래 3줄 구조로 작성한다.**
한 줄은 한 문장, 줄바꿈 없이 작성. 이력서에 그대로 붙여쓸 수 있는 수준으로 작성.

```
[문제] {어떤 기술적 한계/비효율이 있었는지 — 구체적 상황 포함}
[해결] {어떤 기술/방식으로 해결했는지 — 기술 스택과 구현 방식 명시}
[성과] {측정 가능한 결과 또는 개선 효과 — 가능하면 수치 포함, 없으면 정성적 효과}
```

**수치 가이드라인:**
- 이미 적용된 경우: 실제 코드에서 확인 가능한 수치 우선 (스레드 수, 큐 사이즈, TTL 등)
- 추가 적용 가능한 경우: "~% 감소 예상", "~배 향상 가능" 형태로 현실적 예측치 작성
- 수치 근거가 없으면 "응답 시간 단축", "DB 부하 감소" 등 정성적 효과로 대체

---

아래 형식으로 출력한다:

```
## 기술 분석: {입력된 기술/지식}

### 핵심 개념
{이 기술이 어떤 문제를 해결하는지 2~3줄}

---

### ✅ 이미 적용된 것들

#### {번호}. {한 줄 제목} (`파일경로:라인번호`)

[문제] {한 줄}
[해결] {한 줄}
[성과] {한 줄}

(항목이 없으면: "현재 이 기술이 적용된 코드를 찾지 못했습니다.")

```

**현재 적용 수준**: {0~5점} / 5

### 4단계: 사용자 확인

결과 출력 후 아래 질문을 한 번에 묻는다:

```
## 다음 단계

위 분석을 바탕으로 어떻게 진행하시겠습니까?

1. **특정 항목 구현** → 번호를 알려주시면 `/issue`로 이슈를 생성합니다
2. **이력서 문장 다듬기** → 원하는 항목 번호를 알려주시면 A/B/C 3가지 버전으로 다듬어 드립니다
3. **이력서 + 구현 연결** → `/resume-feature`로 이력서 항목을 이슈까지 연결합니다
4. **그냥 참고** → 분석 결과를 참고용으로만 활용합니다
```

### 5단계: 이력서 문장 다듬기 (요청 시)

사용자가 특정 항목의 이력서 문장 다듬기를 요청하면:

선택한 항목의 [문제]/[해결]/[성과]를 기반으로 3가지 버전 제안:

```
[버전 A - 기술 중심]
[문제] {기술적 상황/한계 강조}
[해결] {기술 스택과 구현 방식 강조}
[성과] {수치 포함 결과}

[버전 B - 성과 중심]
[문제] {한 줄}
[해결] {한 줄}
[성과] {수치와 비즈니스 임팩트 강조}

[버전 C - 문제 해결 중심]
[문제] {근본 원인까지 파고든 문제 정의}
[해결] {문제 원인에 직접 대응한 해결 방식}
[성과] {수치 포함 결과}
```

---

### 6단계: 이슈 생성 (요청 시)

사용자가 특정 항목 구현을 요청하면 `/issue` 워크플로와 동일하게 진행:

**불명확한 항목만** 추가로 질문:
- **대상**: 누가 사용하나? (USER / ADMIN / DORMITORY)
- **범위**: 이번 이슈에 포함할 것 / 제외할 것

작업 목록 초안 제안 후 확인받고, GitHub MCP로 이슈 생성:
- owner: Teach-D
- repo: appcenter-16.5-dormitory-project
- title: `[{type}] {기능 설명}` (예: `[refactor] Redis 캐싱 groupOrder 조회 적용`)
- body: 개요 + 작업 목록 + (필요 시) API 설계

이슈 생성 후 `.claude/issues.md` 업데이트.
