# Feature Spec Generator

기능 설명을 받아 UniDorm 프로젝트 구조에 맞는 기획서를 작성한다.

결과는 **채팅창에만** 출력한다. 파일을 생성하거나 수정하지 않는다.

---

## Step 1 — 프로젝트 문서 읽기

기획서를 작성하기 전에 아래 문서를 읽어 현재 프로젝트 구조를 파악한다:

- `docs/api-spec.md` — 기존 API 엔드포인트 패턴, 권한, 응답 형식
- `docs/db-schema.md` — 기존 테이블 구조, FK 관계, 컬럼 명명 규칙
- `docs/architecture.md` — 기능 흐름, 외부 시스템 연동 패턴 (FCM, Redis, 스케줄러 등)

---

## Step 2 — 정보 수집

문서를 읽은 뒤, 기능 설명에서 **유추할 수 없는 것들만** 한 번에 질문한다.

항상 확인해야 할 것:
- **대상 사용자**: USER / ADMIN / DORMITORY 중 누가 이 기능을 사용하는가?
- **비즈니스 규칙**: 조건, 트리거 시점, 예외 케이스 등 모호한 부분

유추 가능한 것은 묻지 않는다.
- 예: "FCM 써야 해"라고 했으면 → FCM 사용 방식은 docs/architecture.md를 참고해 유추
- 예: "쿠폰 만료 3일 전 알림"이라고 했으면 → 스케줄러 필요 여부는 자동 판단

---

## Step 3 — 기획서 작성

수집한 정보와 프로젝트 문서를 바탕으로 아래 형식으로 기획서를 출력한다.

````
# [기능명] 기획서

## 1. 기능 개요 / 목적

- **기능 설명**: 무엇을 하는 기능인지
- **목적**: 왜 필요한지
- **대상 사용자**: USER / ADMIN / DORMITORY
- **관련 도메인**: 기존 도메인 중 해당하는 것 (없으면 신규 도메인명)

---

## 2. API 엔드포인트 설계

### Endpoints

| Method | URL | 권한 | 설명 |
|--------|-----|------|------|
| POST   | /api/... | [AUTH] | ... |

> API 엔드포인트가 필요하지 않은 기능(예: 순수 스케줄러 작업)이면 이 섹션을 생략한다.

---

## 3. DB 스키마 변경사항

### 신규 테이블

| 컬럼명 | 타입 | 제약조건 | 설명 |
|--------|------|----------|------|
| id | BIGINT | PK, AUTO_INCREMENT | |
| ... | ... | ... | ... |

### 기존 테이블 변경

- `테이블명`: 추가/변경할 컬럼 및 이유

### Flyway 마이그레이션

- 파일명: `V{N}__설명.sql`
- 주요 DDL 내용 요약

> DB 변경이 없으면 "변경 없음"으로 표기한다.

---

## 4. 기술적 고려사항

- **외부 시스템 연동**: FCM, Redis, Oracle DB, Selenium 등 사용 여부와 방식
- **비동기 / 스케줄러**: @Async, @Scheduled 필요 여부
- **예외 처리**: 발생 가능한 예외와 대응 방식
- **성능**: 인덱스 추가, 캐싱, 페이징 등 고려사항
- **보안**: 권한 검증, 본인 확인 등 필요 여부
````

---

## 출력 후 안내

기획서 출력 후 다음을 안내한다:
1. 수정이 필요한 부분이 있으면 말해달라
2. 기획서 확정 후 `/api-spec` 스킬로 API 명세서를, `/feature` 스킬로 TDD 구현을 바로 시작할 수 있다
