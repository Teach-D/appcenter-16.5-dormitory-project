# TDD Review Pipeline — 테스트 → 구현 → 보안감사 → 코드리뷰

당신은 UniDorm 4단계 품질 파이프라인의 **오케스트레이터**입니다. 아래 단계를 순서대로 실행하며, **STEP 1에서 반드시 멈추고 사용자 승인**을 받은 후 STEP 2로 진행합니다.

---

## STEP 1 — TDD 테스트 작성 (사용자 승인 필수)

`tdd-test-writer` 서브에이전트를 호출합니다. 아래 정보를 전달합니다:
- 사용자가 제공한 기능 설명 또는 기획서 원문
- 현재 브랜치명 (`git branch --show-current` 결과)

에이전트가 테스트 케이스 목록을 제시하면 **반드시 멈추고** 아래 형식으로 사용자에게 제시합니다:

```
---
[STEP 1/4] 테스트 코드 작성이 완료되었습니다.

작성된 파일:
  {에이전트가 반환한 파일 목록}

커버된 테스트 시나리오:
  {에이전트가 반환한 시나리오 목록}

테스트가 의도한 대로 작성되었나요?
수정이 필요하면 알려주세요. 승인하려면 "OK" 또는 "진행해"를 입력하세요.
---
```

**사용자가 승인("OK", "좋아", "진행해", "approve", "승인")을 보내기 전까지 STEP 2로 절대 진행하지 않습니다.**

수정 요청이 있으면 `tdd-test-writer`를 다시 호출해 수정하고 재제시합니다.

에이전트 반환값에서 아래를 추출해 기억합니다:
- `test_files` — 작성된 테스트 파일 경로 목록
- `br_coverage` — 커버된 BR 번호 목록

---

## STEP 2 — 기능 구현 (승인 후에만)

`feature-implementer` 서브에이전트를 호출합니다. 아래 정보를 전달합니다:
- 사용자가 제공한 기능 설명 또는 기획서 원문
- STEP 1에서 작성된 테스트 파일 경로 목록 (`test_files`)
- 현재 브랜치명

에이전트가 `IMPLEMENTER_RESULT` 블록을 반환하면 아래 값을 추출해 기억합니다:
- `status` — `SUCCESS` 또는 `PARTIAL_FAILURE`
- `files_written` — 작성/수정된 프로덕션 파일 경로 목록
- `tests_passed` — 통과한 테스트 수
- `changes_summary` — 변경 요약

`status: PARTIAL_FAILURE`이면 실패 내용을 사용자에게 보여주고 계속 진행할지 묻습니다.

STEP 2 완료 후:
```
[STEP 2/4] 기능 구현이 완료되었습니다.

테스트 결과: {tests_passed}
작성/수정된 파일: {files_written 목록}

보안 감사(STEP 3)를 시작합니다...
```

---

## STEP 3 — 보안 감사

`secure-reviewer` 서브에이전트를 호출합니다. 아래 정보를 전달합니다:
- STEP 2의 `files_written` 목록
- "위 파일들에 대해 보안 감사를 수행해주세요"

에이전트가 보안 감사 보고서를 반환하면 사용자에게 전달하고 아래를 확인합니다:

```
[STEP 3/4] 보안 감사가 완료되었습니다.

{보안 감사 보고서 전문}

---
Critical 또는 High 이슈가 있습니까? (보안 점수: {통과/조건부 통과/실패})

수정이 필요하면 알려주세요 (feature-implementer로 수정 후 보안 감사를 재실행합니다).
수정 없이 계속하려면 "계속" 또는 "다음"을 입력하세요.
```

**Critical 또는 High 이슈가 있고 사용자가 수정을 요청하면:**
1. `feature-implementer`를 호출해 보안 이슈를 수정합니다
2. `secure-reviewer`를 다시 호출해 재감사합니다
3. 최대 2회 반복 후에도 미해결이면 사용자에게 알립니다

---

## STEP 4 — 코드 리뷰

`code-reviewer` 서브에이전트를 호출합니다. 에이전트는 `git diff HEAD~1` 기준으로 최근 변경사항을 자체 파악합니다.

에이전트가 코드 리뷰 보고서를 반환하면 사용자에게 전달합니다:

```
[STEP 4/4] 코드 리뷰가 완료되었습니다.

{코드 리뷰 보고서 전문}

---
```

코드 리뷰 결과를 보여준 후 파이프라인 완료 요약을 출력합니다:

```
== TDD Review Pipeline 완료 ==

STEP 1 — 테스트 작성: 완료 ({test_files 파일 수}개 파일)
STEP 2 — 기능 구현: {status} ({tests_passed} 통과)
STEP 3 — 보안 감사: {통과/조건부 통과/실패}
STEP 4 — 코드 리뷰: Critical {n} / High {n} / Medium {n} / Low {n}

다음 단계: 이슈를 수정한 뒤 /ship 으로 커밋·푸시·PR을 생성하세요.
```

---

## 에러 처리

| 상황 | 대응 |
|------|------|
| `docs/api-spec.md` 또는 `docs/domain-model.md` 누락 | STEP 1 전에 사용자에게 알리고 중단 |
| 테스트 3회 이상 실패 | 실패 내용 보고 후 계속 진행 여부 확인 |
| 보안 감사 Critical 이슈 수정 2회 실패 | 미해결 이슈 목록과 함께 사용자에게 수동 처리 안내 |
| feature-implementer `PARTIAL_FAILURE` | 실패 내용 보고 후 STEP 3으로 계속 진행할지 확인 |
